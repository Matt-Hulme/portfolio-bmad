## Dev Notes

### Architecture Context

**Data Strategy** [Source: Epic 1]
- Start with static mock data for rapid UI development
- TypeScript interfaces will remain stable when migrating to backend API
- Data structure should support future pagination and search
- Keep data simple but realistic

**Type System Approach**
- Use string literal unions for constrained types (Technology, Role, Industry)
- This provides autocomplete and type safety
- Easy to extend with new values
- Alternative: Could use enums, but string literals are more idiomatic

**Mock Data Diversity** [Source: AC 4-5]
- Must include variety of project types to test UI edge cases
- Some projects with links, some without
- Some with images, some without
- Different role combinations (single role vs. multiple)
- Different technology stacks

### Helper Functions

```typescript
// src/data/projects.ts (continued)

export function getAllProjects(): Project[] {
  return projects;
}

export function getProjectBySlug(slug: string): Project | undefined {
  return projects.find(p => p.slug === slug);
}

export function getUniqueTechnologies(): Technology[] {
  const techs = new Set<Technology>();
  projects.forEach(p => p.technologies.forEach(t => techs.add(t)));
  return Array.from(techs).sort();
}

export function getUniqueRoles(): Role[] {
  const roles = new Set<Role>();
  projects.forEach(p => p.roles.forEach(r => roles.add(r)));
  return Array.from(roles).sort();
}

export function filterProjects(
  filters?: { technology?: Technology; role?: Role }
): Project[] {
  if (!filters) return projects;

  return projects.filter(project => {
    const matchesTech = !filters.technology ||
      project.technologies.includes(filters.technology);
    const matchesRole = !filters.role ||
      project.roles.includes(filters.role);
    return matchesTech && matchesRole;
  });
}
```
